Claude vs. Nutrient ICR
Handwriting Transcription Comparison

A head-to-head comparison of the Nutrient SDK's ICR engine against a general-purpose VLM (Anthropic Claude) on three representative handwriting samples: a structured print recipe card, a cursive recipe on aged paper, and a modern cursive letter.

Date: 2026-05-28 SDK: nutrient-sdk 1.0.6 · nutrient-sdk-native 1.0.6 VLM: Claude Opus 4.7 (native vision)
Headline. On all three samples — including the print-handwriting recipe card that should have been ICR's strongest case — Claude produced complete, downstream-usable transcriptions, while ICR produced fragmentary or unrecoverable text. ICR's reported confidence (0.59 – 0.71) did not correlate with output quality. The boundary between "ICR works" and "ICR doesn't" appears closer to rigid form-field layout vs. anything else than print vs. cursive.

Methodology

  • Three images chosen to span the handwriting difficulty range: clean print on a structured recipe card; aged cursive recipe; modern cursive letter.
  • ICR invoked through this project's /api/extraction/icr endpoint — Vision.extract_content() with VisionEngine.ICR, VisionFeatures.ALL minus the FORM bit. The fullText field of the response is reproduced verbatim below.
  • Claude received the raw image with the implicit instruction "transcribe this handwriting." Output is reproduced verbatim. Reading notes accompany each transcription.
  • No post-processing applied to either output. Neither engine was given any prior context about the document.

Sample 1 — Heavenly Hamburgers Print · Structured Card

This is the case ICR is supposed to be good at: clean, block-style print handwriting on a well-structured recipe card with clear field boundaries.

Observations

  • Even on what should be ICR's strongest case — clear print handwriting in structured fields — Claude transcribes the recipe end-to-end while ICR returns fragmentary, mostly unusable text. A cook cannot make these hamburgers from the ICR output.
  • ICR did catch some structural anchors: "Salt, pepper, Onion satt (salt) er dried chonged (chopped) onions", "Make ints (into) thick pathes (patties)", "Bake Abr. (1hr.) at 350 F", "Sauce:", "from lela to", "Bake for", "Recipe from the kituhgn (kitchen) of:". The skeleton is recognizable.
  • But the structured fields ICR is supposed to handle — quantities, units, ingredient names — are mangled or absent. The ingredient list is not extractable. The sauce sub-recipe is not extractable. Only the section labels survive.
  • ICR appears to have been confused by decorative pre-printed elements (the cheese illustration, the "Here's what's cookin':" red text, the lined rules). Phantom "x 1 ~" at the top and a misread "Reeipe from the kituhgn of:" at the end likely come from the decorative graphics rather than the handwriting itself.
  • Conclusion: in a controlled form context with hand-printed values in pre-defined boxes, ICR works (per the employment-application case in the companion engine-quality doc); on a less structured but still print and well-organized recipe card, it produces output unfit for downstream use.

Claude — transcription Complete

Observations

  • Claude produced a usable, kitchen-ready transcription: every ingredient with quantity and unit, every method step, fractional measurements preserved.
  • ICR's output is not a degraded version of the correct text — it is structurally different content. Most "words" are invented. The fragments that do match the original ("Cream", "egas" → eggs, "milh" → milk, "prepared kin" → prepared tin, "hrs") are scattered among gibberish and provide no reliable anchor for post-processing.
  • ICR's reported confidence on this sample was 0.64 — moderate, not low. The score does not flag the output as unreliable.

What about tuning ICR?

Before concluding, we exercised the obvious knobs the SDK exposes for ICR. None closed the gap.

  • Confidence thresholds. SegmenterSettings.confidence_threshold × WordsDetectionSettings.confidence_threshold swept across nine combinations (0.5 / 0.7 / 0.85 each) on the Heavenly Hamburgers sample. Every combination produced byte-identical output — 29 elements, 0.59 avg confidence, same 324-char text including the phantom "x 1 ~" prefix. The documented confidence knobs do not gate ICR's final output.
  • VLM word refining. HandwritingSettings.word_refining_method was switched from HEURISTIC (default) to VLM with Claude configured. The VLM path did run (latency dropped from 15.5s to 7.2s) but the text output was byte-identical to heuristic mode. Word-refining is downstream of segmentation and can't fix upstream misreads.
  • Deskew. Tested on the tilted Dear Magnus letter. Default 15° tolerance and raised tolerances (45°, 60°) gave identical output. Disabling deskew produced higher reported confidence (0.81 vs 0.71) but the words came back scrambled in spatial fragments. Deskew restores reading order, not word-level accuracy.

There is no combination of documented settings that closes the gap to Claude on the test samples. The recognition model itself is not configurable; only the surrounding pipeline.

Practical implications

  1. For real-world handwriting transcription today — print or cursive — a general-purpose VLM dominates Nutrient ICR. Customers with this workload should not be steered to ICR. The print-handwriting case was expected to be ICR's strength; the structured recipe card shows that "expected strength" still produces output that's not extractable for downstream use.
  2. ICR's reported confidence does not signal output quality. Confidence ranged from 0.59 to 0.71 across the three samples, with no correlation to whether the output was usable. Building downstream logic that thresholds on ICR confidence (e.g., "auto-accept if conf > 0.7") will silently approve unusable content.
  3. The gap is not closeable with prompt engineering or post-processing on the ICR side. The output is not a noisy version of the truth that could be cleaned up — it is largely invented content. Any "fix" would need to happen at the model layer (retraining, swapping the backend, or routing inputs through a VLM).
  4. ICR's clean-success window is narrower than the API name suggests. The companion engine-quality doc shows ICR working well on a clean print employment-application form. The "Heavenly Hamburgers" card is also print handwriting on a structured layout — yet ICR fails on it because the card has decorative graphics, lined rules, and free-form positioning. The boundary between "ICR works" and "ICR doesn't" is not "print vs. cursive" but something closer to rigid form field vs. anything else.
  5. The SDK could close this gap by adding a Vision.transcribe()-style call that uses a configured VLM provider. The Anthropic / OpenAI / Gemini APIs are already exposed by the SDK for describe(); reusing that plumbing with a transcription prompt would give customers a high-quality path through the SDK rather than around it.

Caveats

  • Single engineer, three images. Treat the conclusions as directional, not statistically rigorous.
  • Claude's transcription was performed without an explicit "be conservative / mark uncertainty" prompt. With such prompting, lower-confidence sections would be flagged rather than asserted.
  • This comparison did not test the case where ICR is strong: clean print handwriting in known, rigid form layouts. See the companion document docs/sdk-feedback/2026-05-28-icr-engine-quality.md in the same repository for that side of the picture.
